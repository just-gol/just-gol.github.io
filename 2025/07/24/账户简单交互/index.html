Solana 账户模型概览

Solana 没有以太坊中那种“拥有逻辑的合约账户”，而是将数据和逻辑分离：

类型 说明
账户（Account） 存储状态的数据容器，没有代码逻辑。
程序（Program） 纯逻辑代码，是只读的（不可更改的代码），负责操作账户数据。

🧾 #[account] pub struct Xxx 是程序内部定义的账户数据结构说明书，它对应链上某个账户的 data 字节布局；

🧠 程序（Program）是部署在 Solana 上的智能合约，控制对账户数据的读写，是 executable = true 的特殊账户。

账户的结构

  • owner 决定了哪个程序可以修改这个账户的 data 字段。
  • data 是程序使用 #[account] 定义的结构体序列化后的内容。
  • executable 为 true 表示这是一个程序账户（部署的智能合约代码）,false 表示该账户是“非程序账户”，只是用来存数据或持币的，不包含代码，不能执行调用。
  • 所有账户都需付租金，保持活跃状态；除非余额足够租金豁免

账户的分类

系统账户（System Account）

这是用户的钱包账户，由系统程序创建，默认 owner 是 SystemProgram。

特点
  • 只有 lamports（余额）
  • 没有自定义数据
  • executable = false
  • 只能转账，不支持存结构体

程序账户（Program Account）

这是你部署的智能合约（用 Rust 写的程序），是 executable = true 的账户。

特点
  • 只读，不可修改
  • 存的是 BPF 字节码（程序逻辑）
  • 一旦部署无法更改（immutable）
  • 所有权为 BPFLoader

类型 描述

数据账户 用于存储状态数据，一般由某个程序 init 创建。

实际开发中常见场景

场景 使用账户类型
用户签名调用程序 系统账户（钱包）
部署程序 程序账户
存储用户数据 数据账户 or PDA
给用户自动分配一个专属地址 PDA 账户（由程序推导）

账户操作

Solana 程序无法创建账户或操作 SOL，必须通过 system_program 或其他内置程序来完成：

  • 创建账户：system_program::create_account
  • 分配 lamports：system_program::transfer
  • 分配数据空间：在创建账户时一并指定 space

Solana 程序与账户交互机制（ctx）

Solana 的程序无法主动操作链上账户，它们只能处理被“传入”的账户：

  • #[account(mut)] 表示此账户可变，会被程序修改。
  • Signer<'info> 是调用者的钱包签名者。
  • Program<'info, System> 是内置系统程序。

PDA（Program Derived Addresses）

PDA 就是程序专属的“自动生成的账户”，只能程序能用、用户打不开，非常适合做：用户数据、配置数据、资源分配等。

Solana 中一个程序不能拥有私钥，但可以通过 PDA 创建“它能控制的账户”

特点：

  • 无私钥（由程序推导），不可被任何人直接签名。
  • 程序内部可通过 invoke_signed 操作 PDA。

PDA的应用场景

你要做的事情 要不要用 PDA？
给每个用户一个账户（唯一） ✅ 非常适合
建一个程序自己的配置仓库 ✅ 必须
自动生成地址，避免冲突 ✅ 这是 PDA 最强的地方
不希望用户自己动数据 ✅ 因为 PDA 没私钥
想让程序能随时找到账户 ✅ 公式可推导，永远能找回

账户生命周期

  • 创建： 由 system_program::create_account 创建，指定 owner、space、lamports。
  • 初始化： 由程序使用 #[account(init)] 等方式赋值。
  • 使用： 程序读取或修改数据。
  • 关闭（可选）： 将剩余 lamports 转给其他账户，释放数据空间。
