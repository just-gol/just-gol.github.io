Box堆对象分配

Box 的使用场景

  • 特意的将数据分配在堆上
  • 数据较大时，又不想在转移所有权时进行数据拷贝
  • 类型的大小在编译期无法确定，但是我们又需要固定大小的类型时
  • 特征对象，用于说明对象实现了一个特征，而不是某个特定的类型

使用Box将数据存储在堆上

Box<T> 是一个智能指针，用来 在堆上分配一块内存来存放 T 类型的数据，本身只占用一个指针大小的栈空间

为什么需要 Box<T>

在 Rust 中：

  • 普通变量（如 let x = 5;）是存储在栈（stack）上的；
  • 栈大小有限，如果存储很大的数据（如递归结构、超大数组），会造成栈溢出；
  • 此时，我们就用 Box<T> 把它放到 堆（heap） 上。
  • Box::new(5) 把 5 放到堆上；
  • b 是一个指向堆上 i32 的指针，大小是固定的；
  • 你可以像普通变量一样使用它，因为它实现了 Deref trait，自动解引用。
应用场景
  1. 存储大数据，避免栈溢出
  1. 递归数据结构（如链表、树）

递归类型的大小在编译期是无法确定的，而 Box<T> 可以让结构拥有确定大小：

内存布局示意图

避免栈上数据的拷贝

栈上数据转移所有权时，实际上是把数据拷贝了一份，最终新旧变量各自拥有不同的数据，因此所有权并未转移。

而堆上则不然，底层数据并不会被拷贝，转移所有权仅仅是复制一份栈中的指针，再将新的指针赋予新的变量，然后让拥有旧指针的变量失效，最终完成了所有权的转移

将动态大小类型变为 Sized 固定大小类型

Rust 需要在编译时知道类型占用多少空间，如果一种类型在编译时无法知道具体的大小，那么被称为动态大小类型 DST。

DST 本身不能单独用，要通过胖指针类型包装，让它变成 Sized 类型。

什么是动态大小类型（DST）

动态大小类型就是：在编译时无法确定大小的类型，常见的有：

  • str：字符串切片类型（不是 String，是 &str 或 str 本身）
  • [T]：切片类型，比如 [i32]
  • trait 对象，如 dyn Trait

这类类型 不能单独使用，因为编译器不知道它占用多少空间。

这是非法的，因为 str 是 DST，大小未知，不能单独分配在栈上。

如何让 DST 变为 Sized
  1. 使用 &T 或 &mut T（借用）
  • &str 是一个胖指针，里面包含：
  • 数据的指针
  • 数据的长度信息
  • 所以 &str 是 **Sized**** 的**，即固定大小（2个 word）
  1. 使用 Box<T>（在堆上分配）
  • Box<str> 也是胖指针，是 Sized 的；
  • 虽然 str 是 DST，但 Box<str> 本身是固定大小，在栈上存了一个堆指针 + 长度信息。
  1. 使用 Rc<T> / Arc<T> 包裹 DST

同样原理，Rc<str> 是固定大小，包含指针和元数据。

  1. 使用 trait 对象 dyn Trait
  • dyn Trait 本身是 DST（因为不同实现大小不同）；
  • Box<dyn Trait> 是 Sized，可以作为函数参数、变量等使用。

Deref 解引用

Deref trait 允许你通过 * 运算符或者点操作（.）自动访问智能指针里指向的数据。

这里的 *y 背后其实调用了：

Deref 的定义

Deref trait 允许你通过 * 运算符或者点操作（.）自动访问智能指针里指向的数据。

  • Target 是你要“解引用”得到的实际类型；
  • 返回的是 &Target（不可变引用）；
  • 所以：实现了 Deref 的类型就可以使用 * 运算符来“解开”。

为什么我们需要 Deref

因为 Rust 的点号语法 .method()、.field 和解引用 * 默认只支持裸指针 &T。

如果你定义了智能指针，比如 Box<T>，你需要告诉编译器“我可以当成引用使用”，这就靠 Deref。

点操作符自动解引用

这背后会自动调用多次 .deref()，直到转换为 &str 为止。

* 解引用操作

三种Deref转换

Rust 的自动解引用（Deref coercion）依赖你是否实现了：

  • Deref：用于获取不可变引用（&T → &U）
  • DerefMut：用于获取可变引用（&mut T → &mut U）
三种 Deref 转换规则详解
  1. T: Deref<Target=U>

⟹ 可以将 &T 自动转换为 &U

这是最常见的，也是默认使用 Deref 的方式。

自动应用 .deref() → &T → &U

  1. T: DerefMut<Target=U>

⟹ 可以将 &mut T 自动转换为 &mut U

这发生在你对值进行可变访问时（比如修改内容）

自动调用 .deref_mut() → &mut T → &mut U

  1. T: Deref<Target=U>

⟹ 可以将 &mut T 自动转换为 &U

这是 Rust 为了灵活性支持的规则：

即使你只有 &mut T，如果 T 实现了 Deref，你也可以只读访问它内部的值。

即使是 &mut s，也可以只读地传给 &str 类型参数！

因为 &mut T 是一种 更强权限 的引用，它也可以被当成 &T 来用。

场景 要求 实际发生的转换
&T → &U T: Deref<Target = U> 调用 deref()
&mut T → &mut U T: DerefMut<Target = U> 调用 deref_mut()
&mut T → &U T: Deref<Target = U> 调用 deref()，只读访问

Drop释放资源

Drop 是 Rust 的析构器特征，用于在变量离开作用域时自动释放资源。

Drop trait 的定义

  • 当一个变量 生命周期结束（出作用域） 时，Rust 会自动调用这个 drop() 方法；
  • 通常你不手动调用它，由编译器自动插入调用点；
  • 每个值在 Rust 中只能被 drop 一次。

自定义 Drop 实现

允许手动调用 .drop()

Rust 禁止这样写，是为了防止资源在生命周期未结束前就被重复释放。

正确的提前释放方式

使用 std::mem::drop

标准库提供的 drop(x) 函数可以让你手动释放资源，但变量此后不能再使用。

总结

实现 Drop，就能在变量生命周期结束时做清理；
不允许你自己 .drop()，但可以用 std::mem::drop(x) 提前释放。

Drop 使用场景

对于 Drop 而言，主要有两个功能：

  • 回收内存资源
  • 执行一些收尾工作

Rc 与 Arc

Rc

Rc<T> 代表 Reference Counted，是 单线程下的共享引用计数指针。

它允许多个变量共享一个数据的所有权，通过引用计数来管理资源的释放时机。引用次数归零时，就代表该数据不再被使用，因此可以被清理释放

解决了什么问题

在 Rust 中：

  • 默认一个值只能有一个所有者；
  • 如果你想让多个变量共享所有权，就需要用 Rc<T>。
特性
特性 是否支持 说明
多个所有者 ✅ 多个变量都可以克隆 Rc 来共享所有权
自动释放资源 ✅ 最后一个 Rc 被释放时，内部数据才会被真正释放
可变访问（内部数据） ❌ 不能直接修改；需要配合 RefCell 实现内部可变性
多线程 ❌ Rc<T> 只能在单线程环境使用，线程不安全
替代方案 ✅ 多线程场景用 Arc<T>，加上原子引用计数
使用场景举例
  1. 构建树结构

树的子节点需要被多个父节点拥有（或反过来），所有权不能是唯一的 —— 这时候就需要 Rc<T>。

  1. 基础用法示例
  • Rc::clone(&a) 不会拷贝数据，而是增加引用计数
  • a, b, c 都拥有这个字符串的所有权
  • 当最后一个所有者作用域结束，才会真正释放内存
  1. 真实场景：构建树状结构
  • node1 被多个节点共享作为“尾节点”
  • 每个节点都用 Rc::clone 来共享它
  1. Rc 的限制不可变引用

❌ 不能可变修改 Rc 内部的数据！

  1. 想修改怎么办？

配合 RefCell<T> 使用：实现内部可变性

  • RefCell 允许在运行时检查可变借用，绕过 Rust 编译器的“单个可变借用”限制；
  • ⚠️ 运行时违反借用规则会 panic！

Arc

Arc 全称是 Atomic Reference Counted，是一个线程安全的引用计数智能指针。

它和 Rc<T> 功能类似，但 Arc<T> 的引用计数是原子操作（atomic），可以在多个线程之间安全共享数据。

主要用途

在多线程环境中，让多个线程共享所有权，又不失安全性。

例如：

  • 多个线程同时读取同一份配置数据；
  • 多个任务共享一份不可变的数据；
  • 配合 Mutex<T>、RwLock<T> 实现线程间共享可变数据。
🆚 与 Rc<T> 的区别
特征 Rc<T> Arc<T>
是否线程安全 ❌ 仅限单线程 ✅ 多线程安全
是否支持共享所有权 ✅ ✅
引用计数机制 非原子（更快） 原子操作（慢一些）
用途 单线程数据共享 多线程数据共享
基本用法：多个线程共享数据
  • rc::new(...) 创建共享数据；
  • Arc::clone(&data) 克隆出一个新的引用计数副本；
  • 所有线程共用同一份堆上的数据；
  • 内部使用原子计数器，线程安全 ✅；
不可变的 Arc 数据不能修改

Arc<T> 适用于 Rust 中的 多线程（thread::spawn），不适用于多进程或异步系统的生命周期管理。

什么时候用 Rc vs Arc

单线程数据共享 ✅ Rc

Cell 和 RefCell

Cell

Cell<T> 是一个允许你在不可变引用下修改其内部值的容器类型，是 Rust 提供的“内部可变性”工具之一。

核心特点
特性 值
可在 &self 下修改数据 ✅ 支持内部可变性
实现机制 按值复制（Copy in/out）
支持多线程 ❌（只能用于单线程）
是否可取内部引用 ❌ 不能返回 &T
使用成本 🚀 非借用检查，运行时无开销（比 RefCell 更轻量）
通过不可变引用修改值
  • x.get()：取值（复制一份）；
  • x.set(val)：设置值（按值替换）；
  • 没有可变借用，没有引用检查，全部通过值拷贝实现 ✅；
Cell<T> 不允许返回引用

正确写法（用 into_inner）

所以 Cell<T> 适用于 Copy 类型 或者临时拿出所有权，不适合复杂的结构引用访问。

RefCell

RefCell<T> 是一个运行时检查借用规则的容器，用于在不可变引用 **&self** 下修改内部值。

它允许你：

  • 在不可变引用中对数据进行可变访问；
  • 动态借用可变或不可变引用，并在运行时检查是否安全。
典型应用场景
  • 单线程程序中，需要多个可变引用但不能通过编译器静态借用检查；
  • 跟 Rc<T> 组合，用于多个所有者中修改共享数据；
  • 想要“临时变通”，逃过编译器严格的 mut 检查。
基本用法
调用方法 返回类型 说明
borrow() Ref<T> 创建一个不可变引用
borrow_mut() RefMut<T> 创建一个可变引用
内部是怎么检查借用规则的

RefCell<T>在运行时维护两个计数器：

  • 可变借用最多只能有 1 个；
  • 不可变借用可以有多个，但不能与可变借用同时存在。

如果违反借用规则，程序将 在运行时 panic！

🆚 RefCell vs Cell
特性 RefCell<T> Cell<T>
能否返回引用 ✅（返回 Ref / RefMut） ❌ 不能返回内部引用
是否运行时检查 ✅ 是 ❌ 否
类型是否必须实现 Copy ❌ 不需要 ✅ 需要或使用 into_inner()
使用场景 复杂结构、引用传递 简单字段、小数据修改
配合 Rc 使用（实现多所有者 + 可变共享）
建议
  • ✅ 在需要但不能静态 borrow 的地方用 RefCell<T>
  • ❌ 不要在多线程中用它（不是线程安全的）
  • ✅ 多线程用 Mutex<T> 或 RwLock<T>
  • ✅ 多所有者配合 Rc<RefCell<T>>
总结

RefCell<T> 是“可以借但要守规则”的容器，
编译期不管你借几次，但运行时会检查：

  • 可变一次，多借崩；
  • 不变可多，一变禁混。

Weak

Weak<T> 是 Rc<T> 的“弱引用版本”：它不会增加引用计数，用于打破 Rc 循环引用，防止内存泄漏。

特点

特征 说明
不增加引用计数 不增加 Rc 的强引用数
不影响资源释放 最后一个 Rc 消失时仍然会释放资源
无法直接访问内容 必须使用 .upgrade() 方法尝试访问
常用于父指针、缓存、观察者 构建图结构、树结构等

例子

  • 子节点必须被“拥有”（用 Rc），否则可能提前释放；
  • 父节点只需“知道”子节点是谁就行，不需要被拥有（用 Weak）；
  • 只要 方向是“强引用向下，弱引用向上”，就不会发生内存泄漏！
Weak::new() 是什么意思

Weak::new() 会返回一个 空的弱引用，相当于 “啥也没有”

什么要用 RefCell::new(…) 包起来

因为 Rust 的结构体要求字段在初始化时都要有值。你又想让 parent 可变（能在创建后填入真实的父节点），那就要：

  • 用 RefCell 包住 Weak，这样才能在后面调用 .borrow_mut() 修改它；
  • 否则你必须在创建时就知道真实的父节点 —— 这在树结构中是不现实的（你往往是先创建子节点，再设置父节点）。
upgrade

弱引用本身不能访问数据，必须通过 .upgrade() 升级为强引用 Rc<T>，才能访问其中的值。
