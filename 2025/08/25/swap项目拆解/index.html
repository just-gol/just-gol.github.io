collect_protocol_fee(提取池子累积的协议费)

collect_fund_fee和collect_protocol_fee 一样

  • protocol_fees 是给协议方（团队、治理）的手续费。
  • fund_fees 给项目的“基金”或者“奖励池”，用于比如给流动性提供者发奖励，或者做社区活动经费。

举例

你去商店买东西：

  • 商店会收一定的服务费（手续费）。
  • 这笔服务费分两部分：
    • 一部分给商店老板（协议费用）
    • 一部分放到“员工奖励基金”或者“商店改善基金”（基金费用）

所以在代码里：

  • protocol_fees_token_0 就是商店老板那部分的钱。
  • fund_fees_token_0 就是放到奖励基金里的钱。

两部分都存在池子里，分别记录，等着有人来“提现”或者分发。

函数签名和参数

  • ctx: Anchor 传入的账户上下文（包含所有账户）
  • init_amount_0 / init_amount_1: 创建者投入的初始 token0 和 token1 数量
  • open_time: 池子的开放时间戳（Unix时间，秒）

第一步：校验 mint 是否受支持

目的：
  • 防止恶意或不合规的 mint 被用于建池（比如包含禁止转账等 extension）
  • is_supported_mint() 会检查：
    • 是标准 SPL token
    • 或是允许的 Token 2022 mint（白名单 + extension 校验）

第二步：是否允许建池

平台方可以通过 AmmConfig 配置项全局关闭建池功能（防止用户私自创建池子）。

第三步：调整开放时间 open_time

确保 open_time 不早于当前时间，避免逻辑错误（如立即开放被跳过）。

第四步：为池创建 2 个 vault 账户

每个 vault 是一个托管账户，用于存放 LP 用户的 token0 / token1。其 owner 是程序 PDA，即不能被用户私自操作。

📌 PDA 构造方式：

并传入 bump 保证地址唯一性。

第五步：创建池状态账户 pool_state

这个账户记录 AMM 的全局状态：如 vault 地址、流动性大小、LP mint、公平开盘时间等。

📌 地址可能是 PDA，也可能是 CLI 指定的 signer 地址。

第六步：初始化 observation 状态（记录价格信息）

用于记录 TWAP（时间加权平均价格）或做 oracle 服务等用途。

第七步：转入初始 token 到 vault

从创建者的 token 账户中转出 init_amount_0 和 init_amount_1 到刚刚创建的 vault 中。

📌 这里实际是调用了 spl_token::transfer CPI。

第八步：从 vault 账户中读取余额，验证初始 token 数量是否合理

平台可能限制初始流动性比例，比如 token0 : token1 必须等值，避免创建不平衡池。

第九步：计算初始 LP token 的数量（流动性）

Uniswap v2 的方式，用 √(token0 * token1) 计算流动性，然后：

  • Mint 出 LP token 给创建者，用于代表其贡献的流动性
  • 锁仓一小部分 LP token，防止流动性完全为 0（安全考虑）

第十步：扣除创建池手续费（如果配置有）

  • 系统从 creator 转账 SOL 到平台收款账户 create_pool_fee
  • 然后调用 sync_native 让该 token account 显示更新后的余额（Token 2022 的特性）

第十一步：初始化 pool_state 内容

把这个池的所有信息初始化写入 pool_state，例如：

  • token_0_vault
  • token_1_vault
  • lp_mint
  • observation_state
  • creator
  • open_time
  • auth_bump

返回成功

总结：整个流程数据流动 & 状态变更

AMM初始化账户顸图

  • authority 是 PDA，由程序生成，控制 vault 和 LP mint。
  • creator 是提供资金并初始化池子的用户。
  • 初始 token0 和 token1 从 creator_token_* 转入 vault。
  • LP token 被 mint 到 creator_lp_token。
  • pool_state 是池子的主状态，包含所有信息。
  • observation_state 是 TWAP 使用的 Oracle 状态。
  • create_pool_fee 是协议收手续费的 TokenAccount（可以是 SOL 或 WSOL）。

问题

token_0_mint为什么比token_1_mint小

token_0 必须比 token_1 小，是为了：

  1. 保证 交易对唯一性（不重复建池）。
  2. 简化逻辑，避免顺序歧义。
  3. 保证池子 PDA/地址生成的一致性。
为什么signer_seeds为什么是token_1_vault

之所以用 token_0_vault 做签名，是因为 它就是被创建的那个账户，系统要求“新账户自己必须确认”

swap_base_input(基于输入的数量进行交换)

用户指定输入 token（amount_in），根据池子储备量、交易费率等计算出能换到的输出 token（amount_out），如果换到的 token 大于等于 minimum_amount_out，则进行 swap 交易并转账。

1. 校验是否允许交换

  • 判断池是否开启了 Swap 权限
  • 当前区块时间是否晚于池的开放时间

2. 扣除输入转账手续费

  • 从用户转进的 token 可能是 token-2022，带转账手续费
  • 实际进池的数量是 amount_in - 手续费

3. 判断输入/输出 vault 是 token0 还是 token1

  • 判断是 token0 -> token1，还是 token1 -> token0 的交易方向
  • 获取池中当前 vault 中 token 的数量，记为 total_input_token_amount 和 total_output_token_amount

4. 调用核心曲线函数

  • 传入：
    • 用户实际转入数量（已减手续费）
    • 池中 token0 和 token1 数量
    • 手续费率
  • 返回计算结果结构体，包含：
    • 实际被交换的 token 数量
    • 交换后池中 token 数量
    • 收取的手续费（protocol + fund）
详解

计算一个基于 输入数量固定（base input）的 swap 操作，考虑：

  • 交易费 (trade_fee)
  • 协议费 (protocol_fee)
  • 基金费 (fund_fee)
  • swap 后的池子新状态（token reserve 改变）
  • 用户能拿到多少输出 token
参数 含义 数值（举例）
source_amount 用户输入 token 数量 1_000_000（即 1e6 单位）
swap_source_amount 池中原本的输入 token（token A）数量 10_000_000
swap_destination_amount 池中原本的输出 token（token B）数量 5_000_000
trade_fee_rate 交易总手续费率（万分制） 30（即 0.3%）
protocol_fee_rate 协议在总手续费中抽成比例 1666（即 16.66%，即 1/6）
fund_fee_rate 基金抽成比例 3333（即 33.33%，即 1/3）
